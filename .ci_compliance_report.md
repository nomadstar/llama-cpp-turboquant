### 3. `tests/test-quantize-fns.cpp` and turbo types

`tests/test-quantize-fns.cpp` **does** iterate over all types:

- `for (int i = 0; i < GGML_TYPE_COUNT; i++)` at `tests/test-quantize-fns.cpp:130-184`

So the new turbo types are automatically covered.

This file needs turbo-specific handling in two places.

#### 3a. Threshold handling is missing for turbo types

Quantization threshold selection at `tests/test-quantize-fns.cpp:147-156` only special-cases `TQ1_0`, `TQ2_0`, some low-bit K/IQ types, and `NVFP4`.

The new turbo types currently fall through to the default max quantization error `0.002f`.

Local repro:

- `./build/bin/test-quantize-fns -v`
- output reaches `Testing turbo3`
- reports `turbo3 absolute quantization error: FAILED (0.031220)`

So yes: turbo types need explicit treatment here, either by:

- adding per-type thresholds for `GGML_TYPE_TURBO2_0`, `GGML_TYPE_TURBO3_0`, `GGML_TYPE_TURBO4_0`, or
- improving the quantizers until they fit the default threshold.

#### 3b. Dot-product buffer sizing is wrong for turbo `vec_dot_type = F32`

`dot_product_error()` allocates:

- `tmp_q1(2*test_size)` at `tests/test-quantize-fns.cpp:87`
- `tmp_q2(2*test_size)` at `tests/test-quantize-fns.cpp:88`

Then it does:

- `const auto * vdot = ggml_get_type_traits_cpu(qfns_cpu->vec_dot_type);` at `tests/test-quantize-fns.cpp:90`
- `vdot->from_float(test_data2, tmp_q2.data(), test_size);` at `tests/test-quantize-fns.cpp:93`

For the turbo types, CPU traits set:

- `GGML_TYPE_TURBO3_0 -> vec_dot_type = GGML_TYPE_F32` at `ggml/src/ggml-cpu/ggml-cpu.c:408-413`
- `GGML_TYPE_TURBO2_0 -> vec_dot_type = GGML_TYPE_F32` at `ggml/src/ggml-cpu/ggml-cpu.c:414-419`
- `GGML_TYPE_TURBO4_0 -> vec_dot_type = GGML_TYPE_F32` at `ggml/src/ggml-cpu/ggml-cpu.c:420-425`

The corresponding dot kernels consume an F32 RHS:

- `ggml_vec_dot_turbo3_0_f32()` at `ggml/src/ggml-cpu/ggml-cpu.c:3384-3401`
- `ggml_vec_dot_turbo2_0_f32()` at `ggml/src/ggml-cpu/ggml-cpu.c:3404-3420`
- `ggml_vec_dot_turbo4_0_f32()` at `ggml/src/ggml-cpu/ggml-cpu.c:3423-3437`

`2*test_size` bytes is not enough for `GGML_TYPE_F32`; that path needs `4*test_size` bytes.

This matches the local repro exactly:

- `turbo3` prints the absolute error failure
- then the process aborts with `double free or corruption (!prev)` before finishing dot-product output

Conclusion:

- Yes, turbo types need test-side handling in `tests/test-quantize-fns.cpp`.
- The immediate correctness bug is the fixed-size temp buffer assumption in `dot_product_error()`.

### 4. `ggml_flash_attn_ext_set_page_table()` bounds handling

`GGML_MAX_SRC` is `10` at `ggml/include/ggml.h:224`.

The new setter is:

- declaration at `ggml/include/ggml.h:2355-2357`
- implementation at `ggml/src/ggml.c:5402-5407`

Current implementation:

- checks `a->op == GGML_OP_FLASH_ATTN_EXT`
- checks `a->src[5] == NULL`
- assigns `a->src[5] = page_table`

It does **not** explicitly assert that index `5` is within `GGML_MAX_SRC`.

Current state is safe because:

- `GGML_MAX_SRC == 10`
- `5 < 10`

But the API does not encode that assumption defensively. For CI compliance and future safety, it should add an explicit guard such as:

- `GGML_ASSERT(5 < GGML_MAX_SRC);`

Optional additional validation would also help:

- `GGML_ASSERT(page_table != NULL);`
- `GGML_ASSERT(page_table->type == GGML_TYPE_I32);`
